# 代码重构

在不改变外部行为的前提下，改善代码结构、可读性和可维护性。重构适合渐进式改进现有代码，而不是推倒重写。

## 何时使用

当出现以下情况时使用本技能：

- 代码难以理解、修改或扩展
- 函数、类或模块承担了过多职责
- 存在重复逻辑、深层嵌套、命名不清等代码异味
- 新功能难以落地，根因在于现有结构不合理
- 用户明确要求清理、整理或重构现有代码

## 核心原则

1. **保持行为不变**：重构只调整结构和表达，不改变功能、输出和对外接口行为。
2. **小步推进**：每次只做一类小修改，降低风险，便于回退和验证。
3. **测试优先**：没有测试保障时，先补充必要验证，再进行结构性调整。
4. **边界清晰**：不要把重构和新功能开发、问题修复混在同一次改动里。
5. **以维护为目标**：优先解决真实的理解成本、重复劳动和扩展阻力，不为“看起来更高级”而改。

## 常见适用操作

- 提取函数、拆分过长函数
- 拆分类或模块，收拢职责边界
- 重命名变量、函数、类型或模块
- 消除重复代码和无效抽象
- 扁平化复杂条件和控制流
- 以明确类型、常量或对象替代含糊表达
- 删除死代码，整理依赖关系

## 不应重构的情况

- 代码稳定且基本不会再演进
- 关键路径代码缺少最基本的验证手段
- 当前目标是紧急交付，而不是改善结构
- 没有明确问题，只是出于主观偏好想“顺手改改”

## 安全流程

1. **识别问题**：确认结构问题具体是什么，以及它如何影响维护或扩展。
2. **建立保护**：确保已有测试，或补充最小必要验证。
3. **小步修改**：一次处理一个问题，避免大范围联动改动。
4. **持续验证**：每一步后检查行为是否保持一致，必要时运行测试和手动验证。
5. **收尾整理**：清理注释、文档和无关残留，确保最终结构清晰。

## 结果标准

- 代码职责更单一，层次更清楚
- 重复更少，命名更准确，控制流更容易理解
- 对后续修改更友好，新增需求不再被旧结构明显阻碍
- 所有既有行为保持不变
