# 代码重构

在不改变外部行为的前提下，改善代码结构、可读性和可维护性。

## 核心原则

**保持行为不变**：只调整结构和表达，不动功能、输出和对外接口。

**小步推进**：一次处理一类问题，降低风险，便于回退。

**测试优先**：没有测试保障时，先补充最小必要验证，再动结构。

**边界清晰**：重构和新功能开发、bug 修复分开做，不混在同一次改动里。

**以维护为目标**：解决真实的理解成本、重复劳动和扩展阻力，不为"看起来更高级"而改。

## 常见操作

- 提取函数、拆分过长函数
- 拆分类或模块，收拢职责边界
- 重命名变量、函数、类型或模块
- 消除重复代码和无效抽象
- 扁平化复杂条件和控制流
- 以明确类型、常量或对象替代含糊表达
- 删除死代码，整理依赖关系

## 执行流程

**第一步：理解问题**
阅读代码，搞清楚用户的具体诉求——是可读性差、职责过多、重复逻辑，还是扩展受阻？如果不明确，先问清楚再动手。

**第二步：列出重构点**
识别主要问题，控制在 3-5 条以内。每条说明是什么问题、影响是什么。范围较大时，先和用户对齐优先级。

**第三步：确认测试保障**
有测试就确认覆盖情况；没有测试就提示用户，或为关键路径补充最小验证，再继续。

**第四步：逐步改写**
按优先级依次处理，每次只改一类问题。每处改动附简短说明，解释为什么这样改。

**第五步：收尾验证**
检查所有既有行为是否保持不变。清理注释残留，确保最终结构清晰。

## 输出格式

- 先用 2-3 句话说明发现的核心问题
- 输出修改后的完整代码块（用户有特别要求时才输出 diff）
- 每个主要改动附一行注释或说明，解释改动原因
- 多个重构点时分步骤展示，每步独立可验证

## 不做的事

- 代码稳定且基本不再演进
- 关键路径缺少最基本的验证手段
- 当前目标是紧急交付
- 没有明确问题，只是主观上想"顺手改改"

遇到这些情况，说明原因，不强行重构。
