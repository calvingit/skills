# 代码简化

在保持功能完整的前提下，改善代码表达，使其更清晰、一致、易读。

## 核心原则

**保持行为不变**：只改写法，不改功能、输出和对外行为。

**优先清晰**：明确易读优于紧凑炫技。压缩到看不懂就过了。

**遵循现有风格**：与项目已有的命名、组织方式和错误处理习惯保持一致。

**只做轻量调整**：聚焦局部表达优化，不主动做模块拆分或大范围结构调整。

## 常见操作

- 简化条件表达式和控制流
- 减少不必要的嵌套和重复判断
- 改善变量、函数、类型或常量命名
- 删除显而易见的冗余注释
- 合并局部重复逻辑，去掉多余中间层

## 执行流程

1. **定位范围**：优先处理当前会话中最近修改或新写的代码
2. **扫描问题**：识别命名模糊、嵌套过深、冗余判断、绕弯写法等
3. **逐点改写**：每处改动保持最小，不扩散到无关部分
4. **输出结果**：给出简化后的完整代码块，附简短改动说明

## 不做的事

- 不改变模块边界或整体架构
- 不拆分类、模块或大函数
- 不引入额外设计模式或抽象层级
- 不把轻量整理扩展成完整重构

遇到超出范围的问题，指出来但不擅自处理——交给 code-refactor。
